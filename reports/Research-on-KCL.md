## KCL 调研

### KCL 的特点

#### 优点

1. 类似 Python 的语法，新手友好。
2. 可以用许多种语言开发插件，支持副作用。
3. KCL 的 validate 功能在校验逻辑简单时是很清晰的。

#### 缺点

1. 作为一门编程语言的文档一般。See [参考手册](https://www.kcl-lang.io/zh-CN/docs/reference/lang/tour)
2. 部分编程语义是独有的：
   - `map` 和 `filter` 是关键字，有约定好的用法，并非高阶函数。
   - 对于配置这样数据驱动的需求，没有 "Everything is an expression." 的特性，表达力相对弱，存在冗余。
   - 虽说有插件系统，但生态差，实际用 KCL 做一些数据外的操作是很难的。

### KCL 的用法

KCL 在企业生态中主要作为各类云原生配置的包装或辅助工具链的一部分, 只是原生 YAML 和用户之间的校验层和数据组合层，一般不承载逻辑。

### 和 ruyipack 的关系

> 下文中，“前端”表示 KCL, CLI 或其他直接与用户交互的层级；
> “中端”表示当前的 Rust 层，负责把前端的输出读入并进行解析和处理；
> “后端”表示各种 Renderer, 负责导出文件。

> ruyipack 的一个必须的后端是 RPM Spec. 其语法比较繁杂，有众多元数据。

在将 KCL 作为 Better YAML 来结构化表示 RPM Spec 的情境下，KCL 的能力是足够的，它可以解决各个简单数据之间的引用关系，替代部分简单变量插入的 RPM Spec Macro 的工作。

但是当需求从 Better YAML 提升到需要在生成 Spec 前就解决部分自定义的宏规则（比如 Build System 约定）时，KCL 的表达力会不够，维护体验和使用体验都不会很好。

有几种可能的解决方案：

1. KCL 只负责承载输入的类型和参数，参数的 apply 和展开在中端进行。
2. 通过 Rust/Go 等 KCL 支持的语言编写插件，直接在 KCL 内解析，后端部分也可以进行这样的改造。

上述方案的缺点都是进行调整时需要触发 Rust 的重编译。

方案一中的 KCL 完全可以被 YAML 替代。KCL 的表达力肯定无法超越通用编程语言，业务复杂时更是如此。YAML 作为前端，由中端负责解析和校验是更合适的，也使得开发更集中，同时能甩掉对 KCL 的依赖。

方案二和方案一实际上并无本质区别，宏展开这个步骤还是在插件内部实现的，维护起来也需要编写对应的语言。

一个通用的解决方案是换一个和 Rust 相比学习曲线更缓的解释型通用编程语言，这样无论是用什么前端都会更加灵活。
